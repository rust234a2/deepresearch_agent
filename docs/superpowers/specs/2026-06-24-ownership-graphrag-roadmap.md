# 股权关系层 + GraphRAG 路线图（模块分解）

日期：2026-06-24

本文件是**程序级路线图**，不是单个模块的 spec。它把"股权关系层 + GraphRAG + 查询编排"分解为 16 个独立模块，确定边界、依赖与顺序。每个模块各自走一遍 设计 → spec → plan → 实现（TDD）循环；本路线图只锁定整体形状，不展开各模块细节。

## 背景

现有 Agent 基于工商登记 + 联系方式 + 经营范围语义检索（`rag/`）。本次引入**股权关系数据**（已清洗的 shareholders / investments 两份边数据，均无统一社会信用代码、仅名称），目标是支持"谁持有/它持有谁/关联供应商"以及多跳股权推理，并加一层查询编排。

## 全局原则与边界

- **确定性优先、零 LLM**，唯一例外是 **C1 查询复杂度检测**，且 C1 带确定性兜底（LLM 不可用→启发式路由）。
- **数据纪律红线**：LLM 与系统**绝不**做采购批准/拒绝/风险结论；已解析企业仍固定 `insufficient_evidence`。LLM 只用于查询复杂度分类这一辅助环节。
- **数据本地化**：C1 只把**用户问题**发给 LLM（不含企业数据），不破坏受限工商数据本地化。
- **名称连接**：股权数据无信用代码，公司↔公司用规范化全名连接（较可靠）；自然人仅名字，连接标"线索级、低置信"，不当已验证控制关系。
- **复用**：GraphRAG 的向量索引（B4）直接复用现有 `rag/` 的 bge 嵌入 + FAISS。
- **缓做**：向库外递归扩展做深多跳、LLM 自然语言生成 + 流式输出，均不在本路线图，后续单独决策。

## 模块分解

### 阶段 A — 股权关系层（确定性，GraphRAG 的地基）

- **A1 清洗模块化**：把 `scripts/clean_qcc_shareholder_data.py`、`scripts/clean_tyc_investment_data.py` 提升为带合成 fixture 的 TDD 模块（对齐 `company_data_cleaning`）。
- **A2 边表入库**：新表 `company_shareholders` + `company_investments`，按规范化名连 `companies`；schema 升 v3；原子构建。
- **A3 Repository 读层**：`get_shareholders(code)` / `get_investments(code)`。
- **A4 关联计算**：共享库外节点连供应商（共同控股股东/母公司/自然人实控人）+ 噪声过滤（托管方、宽基基金）。
- **A5 Agent 接入**：股权邻域 + 关联方包成工具/研究维度。

### 阶段 B — GraphRAG（确定性，建在 A 之上）

- **B1 实体解析 → 节点表**：把两份边里的实体合并去重成规范节点（公司/自然人/基金），稳定 `node_id`，标 `in_database`/`is_person`，名称消歧。
- **B2 图谱构建/加载**：`graph_nodes` + `graph_edges` 表（持股/投资边带比例、状态），可选 NetworkX 内存图，增量重建。
- **B3 图遍历/多跳查询层**：ego-graph、最短路径、共同控制人、向上穿透实控人——纯确定性图算法。
- **B4 节点文本化 + 向量索引**：节点/邻域描述成文本 → 复用 `rag/` 的 bge + FAISS。
- **B5 混合检索**：向量找种子节点 → 沿图扩展多跳邻域 → 组装子图上下文（local/global search）。
- **B6 社区检测 + 摘要/生成（缓做，可能需 LLM）**：社区聚类 + 子图摘要；单独决策是否做、是否用 LLM。
- **B7 GraphRAG Agent 接入**：图检索作为工具/维度，带引用。

### 阶段 C — 查询处理与编排

- **C1 查询复杂度检测（★唯一 LLM，带确定性兜底）**：分析查询特征（是否指名实体、是否问关系/控制/路径、推理跳数）→ 判定 简单/中等/复杂。
- **C2 多策略检索编排**：简单 → 传统检索（现有 `rag/` 语义检索）；复杂 → 图RAG 检索（B5）；中等 → 组合后融合排序。
- **C3 结构化答案生成**：从检索上下文拼出带引用的结构化结果（无 LLM、无流式）。
- **C4 降级与重试**：高级策略失败 → 降级传统检索；传统也失败 → 返回系统异常；重试（非流式）。
